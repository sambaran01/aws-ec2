Production 3-Tier Architecture

Build a scalable, highly available web application infrastructure using AWS best practices. This project covers S3 static hosting, CloudFront CDN, Application Load Balancer, Auto Scaling, EC2 instances, and RDS Multi-AZ database.

S3 CloudFront EC2 RDS Auto Scaling

Overview

A 3-tier architecture separates your application into three logical layers: presentation, application, and data. This separation provides better security, scalability, and maintainability for production systems.

What You Will Learn
  • Design highly available multi-tier architectures
  • Implement CDN for global content delivery
  • Configure auto scaling for dynamic workloads
  • Set up Multi-AZ RDS for database redundancy
  • Secure network architecture with proper subnet design
Engineering Notes

Why this architecture exists: 3-tier architectures provide clear separation of concerns, making systems easier to scale, secure, and maintain. Each tier can be scaled independently based on demand.

When to use it: Use this pattern for web applications that need high availability, global reach, and the ability to handle variable traffic loads.

Tradeoffs: Increased complexity and cost compared to simpler architectures. Requires more operational overhead but provides better reliability and performance.

Real-world usage: This pattern is used by companies like Netflix, Airbnb, and Spotify for their core web applications.

Step 1: Create VPC

Start by creating a Virtual Private Cloud to isolate your infrastructure.

bash 
aws ec2 create-vpc \
  --cidr-block 10.0.0.0/16 \
  --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=production-vpc}]'

This creates a VPC with a /16 CIDR block, providing 65,536 IP addresses for your infrastructure.

Step 2: Create Subnets

Create public and private subnets across multiple availability zones for high availability.

bash 
aws ec2 create-subnet \
  --vpc-id vpc-xxxxx \
  --cidr-block 10.0.1.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=public-subnet-1a}]'

Step 3: Configure Routing

Set up route tables to control traffic flow between subnets and the internet.

bash 
aws ec2 create-route-table \
  --vpc-id vpc-xxxxx \
  --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=public-rt}]'

Step 4: Deploy Servers

Launch EC2 instances with Auto Scaling to handle variable traffic loads.

bash 
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name production-asg \
  --launch-template LaunchTemplateName=web-server-template \
  --min-size 2 \
  --max-size 10 \
  --desired-capacity 2 \
  --vpc-zone-identifier "subnet-xxxxx,subnet-yyyyy"

Step 5: Configure Monitoring

Set up CloudWatch alarms to monitor your infrastructure health and performance.

bash 
aws cloudwatch put-metric-alarm \
  --alarm-name high-cpu-alarm \
  --alarm-description "Alarm when CPU exceeds 70%" \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 300 \
  --threshold 70 \
  --comparison-operator GreaterThanThreshold

Troubleshooting

Common issues and their solutions.

Port Not Accessible

Problem: Cannot connect to EC2 instance on port 80 or 443.

Solution: Check security group rules. Ensure inbound rules allow traffic on ports 80 and 443 from 0.0.0.0/0. Verify NACL rules and route tables.

Permission Denied

Problem: AWS CLI commands fail with "Access Denied" error.

Solution: Verify IAM user has necessary permissions. Check IAM policy attached to user or role. Use aws sts get-caller-identity to verify credentials.

Service Not Starting

Problem: Application fails to start on EC2 instance.

Solution: Check application logs in /var/log/. Verify all dependencies are installed. Check if required ports are already in use with netstat -tulpn.

Database Connection Failed

Problem: Cannot connect to RDS database from EC2.

Solution: Verify RDS security group allows inbound traffic from EC2 security group. Check database endpoint and port. Ensure database is in available state.
